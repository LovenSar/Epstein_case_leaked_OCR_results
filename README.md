# Epstein Leaked Files - LLM Recognition Outputs

## Overview
This repository stores recognition outputs generated from leaked Epstein-case source files.
Outputs are produced by an automated recursive pipeline that routes pages, runs OCR/visual analysis, and writes per-file text results.

## Version Layout
- `*.txt`: Version1 OCR outputs (legacy baseline).
- `*_2.txt`: Version2 OCR outputs (upgraded pipeline).
  - Includes file-level metadata header `DOJ_FILE_META`.
  - Uses `DOJ_CLEAN_V2` tagging in page/frame headers.

## Routing Mode (Version2)
Version2 generation used the fast pipeline profile:
- `--fast-scan`
- `--resume`
- `--scan-cache`
- `--output-suffix "_2.txt"`

### Behavior
- In fast-scan mode, page routing is handled by the fast path (`process_single_image_fast`) rather than multi-vote router loops.
- Heuristic routing emits `DOC` or `VISION` per page.
- In this corpus, V2 page-route distribution is:
  - `DOC`: `26,398` pages (`76.16%`)
  - `VISION`: `8,264` pages (`23.84%`)

### Model Topology (Version2 Defaults)
Default logical model roles in `recursive_llm_processor.py`:

| Role | Default Model | Notes |
|---|---|---|
| `router` | `qwen3.5:9b` | Route decision (`DOC`/`VISION`), text-presence checks |
| `ocr_glm` | `glm-ocr:latest` | High-fidelity OCR for doc path and vision-text supplement |
| `ocr_qwen` | `qwen3.5:9b` | Document OCR branch |
| `desc_qwen` | `qwen3.5:9b` | Vision description branch |
| `refiner` | `qwen3.5:9b` | Final expansion/cleanup (non-fast mode) |

### Routing Logic
- **Normal mode** (`--no-fast-scan`):
  - LLM router votes (`--router-votes`, default `3`) to choose `DOC` vs `VISION`.
  - `DOC` path runs dual OCR (`glm-ocr` + `qwen`) and merges outputs.
  - `VISION` path runs detailed description + text check + optional OCR tail.
  - Refiner stage expands and structures final output.
- **Fast mode** (`--fast-scan`, used for this V2 corpus):
  - Programmatic heuristic routing (`_program_route_image`) with no router vote loop.
  - Single model call per page/image and no refiner stage.
  - Lower cost and higher throughput for full-corpus passes.

### Version1 -> Version2 Workflow Upgrades
Compared with the legacy V1 outputs (`*.txt`), V2 (`*_2.txt`) adds:

| Area | Version1 (legacy baseline) | Version2 |
|---|---|---|
| Output identity | Baseline text outputs | Dedicated suffix `_2.txt` + explicit clean tagging |
| File-level metadata | No standardized `DOJ_FILE_META` marker | `DOJ_FILE_META` marker on every merged file |
| Page/frame tagging | Legacy headers | `[TAG:DOJ_CLEAN_V2]` page/frame tags |
| Routing clarity | Legacy style (often lightweight `FAST` style) | Explicit `DOC`/`VISION` routing with tracked distribution |
| Resume robustness | Basic legacy behavior | State + output header reconciliation + truncated-output detection |
| Scan efficiency | Full directory walk each time | Reusable scan cache (`--scan-cache`) |
| Observability | Limited | Disk logs + per-model event logging + periodic system snapshots |
| Detail density | Legacy baseline | Enrichment controls (`detail_multiplier`, `enrich_from_output`) |

## Runtime Environment
### Hardware (Recommended)
- **Minimum (small batches / CPU-fallback possible):** 8+ CPU cores, 32 GB RAM, SSD/NVMe.
- **Recommended (stable local throughput):** 12+ CPU cores, 64 GB RAM, NVIDIA GPU with 16 GB+ VRAM, NVMe SSD.
- **Large-scale/full-corpus friendly:** 24 GB+ VRAM GPU and 128 GB RAM for better concurrency headroom.

### Software
- Python 3.10+
- Ollama running locally or remotely (default endpoint: `http://localhost:11434/api/generate`)
- Python packages:
  - `requests`
  - `Pillow`
  - `tqdm`
  - `pdf2image`
  - `opencv-python` (for video processing)
- Poppler utilities available to `pdf2image` (for PDF page conversion)

Install Python dependencies:

```bash
python -m pip install requests pillow tqdm pdf2image opencv-python
```

### Ollama Setup
Pull required models:

```bash
ollama pull qwen3.5:9b
ollama pull glm-ocr:latest
```

Start or verify Ollama service:

```bash
# start service (if not already running)
ollama serve

# check service health / loaded models
curl http://localhost:11434/api/tags
```

If Ollama runs on another host, override endpoint with `--ollama-url`.

## Usage
Run from repository root:

```bash
python recursive_llm_processor.py --root "."
```

### Version2-style run example

```bash
python recursive_llm_processor.py \
  --root "." \
  --output-suffix "_2.txt" \
  --resume \
  --scan-cache \
  --fast-scan \
  --show-page-progress
```

### Full routing/refiner mode (higher quality, higher cost)

```bash
python recursive_llm_processor.py \
  --root "." \
  --output-suffix "_hq.txt" \
  --resume \
  --scan-cache \
  --no-fast-scan \
  --router-votes 3 \
  --show-page-progress
```

## Important CLI Options
- `--root`: root directory to scan recursively.
- `--state-file`: resumable progress file.
- `--resume` / `--no-resume`: continue from prior progress or start fresh.
- `--overwrite`: replace existing output files.
- `--output-suffix`: output suffix (e.g., `.txt`, `_2.txt`).
- `--fast-scan` / `--no-fast-scan`: switch between fast profile and full router+refiner profile.
- `--router-votes`: route vote count for non-fast path.
- `--scan-cache`: reuse directory scan cache.
- `--ollama-url`: use local or remote Ollama endpoint.
- `--num-ctx`, `--num-predict`, `--keep-alive`: core Ollama runtime controls.
- `--timeout`, `--retries`: API reliability controls.
